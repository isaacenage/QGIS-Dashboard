# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A QGIS desktop plugin that builds **ArcGIS-Dashboards-style interactive dashboards** from a QGIS project's vector layers. The user adds data-driven tiles (indicators, charts, lists, a live map, selectors) into a dockable panel; selecting a value in one tile **cross-filters every other tile** in real time. Dashboard layout and configuration are saved inside the `.qgz` project file.

The shippable plugin lives in **`qgis_dashboard/`** — a QGIS Plugin Builder scaffold whose generated dialog skeleton was replaced with a real dashboard implementation (a dock widget + a signal bus + pluggable element types). Keep the Plugin Builder structure intact (it is what makes the plugin install cleanly and pass `plugins.qgis.org` validation); add features as new modules within it.

> Historical note: an earlier hand-written prototype lived in `qgisdashboardmvp/`. Its architecture worked but was not Plugin-Builder-shaped, so it failed to install cleanly. It has been folded into `qgis_dashboard/` and removed. Do not recreate a parallel plugin folder.

## Architecture (the big picture)

The dashboard is a **standalone top-level window** (not a dock) holding **multiple pages**. Read these files together — `bus.py`, `window.py`, `page_view.py`, `dashboard_canvas.py`, `theme.py`, `elements/base.py`, `qgis_dashboard.py` — before changing behavior.

1. **The bus / shared context (`bus.py` → `DashboardBus`)** is the one object every element shares. It carries the live `iface` (for the map mirror), the active `Theme` (both **global**), and **page-local** cross-filter state: each page has its own connection graph + per-source filters, keyed by the active page. The window calls `set_active_page(id)` on page switch; `set_filter`/`set_targets`/`combined_filter_for`/`connections_to_dict(page_id)`/`load_connections(data, page_id)` all operate on the active (or given) page. `forget_page(id)` drops a deleted page's state; `forget_element(id)` drops a tile from the active page. (`_source_filters`/`_connections` are properties returning the active page's dicts, so call sites are unchanged.) Cross-filtering follows ArcGIS *source → action → target*, but wiring is **explicit, user-editable, and only links tiles on the same page**. Signals:
   - `filtersChanged()` — any source filter changed; each target recomputes via `combined_filter_for(self.id)` (the AND of every *connected* source's expression, or `None`).
   - `filtersCleared()` — "Clear filter" pressed; sources reset their own selection state (highlighted bar, combo).
   - `connectionsChanged()` — the wiring graph changed.
   - `layersChanged()` — project layers added/removed.
   - `themeChanged()` — global theme changed; every tile re-applies appearance.
   - `featureAction(object)` — list of feature ids to zoom/flash; the map listens.

   Sources call `set_filter(self.id, expr_or_None)`. Wiring lives in `_connections` (`set_targets`, `targets_of`, `sources_of` (reverse lookup), `is_connected`, `set_connected` (toggle one edge), `connections_to_dict`/`load_connections`). `forget_element(id)` drops a removed tile from filters + graph.

2. **The window (`window.py` → `DashboardWindow`)** is a `QMainWindow` shown as its own OS window, with the branded gradient logo as its title-bar icon (`setWindowIcon(logo_icon())`). Chrome is a **left icon rail** (`sidebar.py → Sidebar`), not a horizontal toolbar: a slim 56px column of icon-only `QToolButton`s (tinted SVG glyphs from `icons.py`, tooltips double as accessible names) exposing `Home` · `Add element`, `Add page` · `Zoom in/out/Reset` · `Clear filter`, then `Save` and a `Settings` gear together at the **foot** of the rail, grouped by thin dividers. `Home` (→ `show_start`) returns to the Start screen; `Save` (→ `save_to_file`) writes the dashboard to a portable `.qdash` file. **`Add element` is an icon picker, not a form** — it opens `element_picker.py → ElementPicker` (→ `open_element_picker`), a **slim left-docked overlay panel** that mirrors the `InspectorPanel` chrome (flat, full-height, a soft hairline `border-right`, a "Add element" title + ✕ header) but is **thinner** and pinned **flush against the right edge of the rail** with no gap (an in-window child of the central widget, raised over the canvas, re-pinned in `resizeEvent`, slides out from behind the rail). It shows one tinted `el_<type_name>` glyph per element type (from `icons.py`); hovering a glyph reveals its name as a tooltip. `open_beside(rail)` **toggles** it (the rail button reopens/closes); the ✕ or picking a glyph closes it. Picking a glyph emits `elementChosen(type_name)` → `_on_element_chosen` → `add_element(type_name, default_config)` — there are **no** type/title/layer/field inputs at add time; the tile is added with a friendly default title and configured afterward from its right-click `Configure…` menu. **Theming is not a direct rail button** — the `Settings` gear opens `settings_dialog.py → SettingsDialog`, a **two-column master/detail** hub: a slim left nav (`QListWidget`) lists sections (*Themes* / *Layout* / *About*) and the right `QStackedWidget` shows that section's controls. **Themes vs Layout is a deliberate split: *Themes* owns the visual *skin* (colors, chart palette, fonts), *Layout* owns *geometry* (corner radius, spacing, text sizes) — so nothing is duplicated across the two.** *Themes* **embeds the whole-dashboard theme editor inline** (`appearance_dialog.py → AppearanceForm` in `mode="global"`, applied live via `on_appearance`=`_apply_global_theme` — no separate dialog hop; the wrapper preserves the *live* Layout-owned metrics so editing a color never reverts a size/radius the user just changed). In `mode="global"` the form shows **only** colors + chart palette + body/heading **font families** — the text-size spinners and corner-radius spin are gone (they moved to Layout); `mode="element"` (the per-tile override) still exposes per-tile sizes. *Layout* holds, top to bottom: the global **canvas-size** control (a preset combo — 16:9, A4, Letter, … — + width/height px spinners; `on_canvas_size` → `_set_canvas_size` → every page's `DashboardCanvas.set_region`, then reframes the current view), the global **corner-radius** slider (`on_radius` → `_set_global_radius`), then the **element-gap / spacing** slider (`on_gap` → `_set_global_gap` → every page's `DashboardCanvas.set_gap`), then three **text-size** spinners — base / element-title / indicator-value (`on_size(key, value)` → `_set_global_text_size`, each updating one of `font_size`/`title_size`/`value_size`). *About* is the story panel; declutter candidates land here rather than on the always-visible rail. **The dashboard theme is strictly canvas-scoped — fonts AND colors.** The theme styles only the **canvas tiles** (re-applied per-tile by `Theme.tile_qss`, which re-themes labels, titles, the indicator value, secondary text, embedded tables/lists and inline inputs) plus the canvas drawing-area background (`#dashPageView`/`#dashCanvas` → `window_bg`). Everything else — the **plugin chrome** (left rail and its icons, page tab bar, dialogs, the Settings hub, the inspector panel `side_panel.py`, the element picker `element_picker.py`, status bar, menus, tooltips) — is locked to **two fixed constants in `theme.py` that the theme can never change**: `SYSTEM_FONT_STACK` (Century Gothic + fallbacks, wired into `window_qss`'s universal `* { font-family: … }`) and the `CHROME` palette dict (the neutral light look). Mechanically, `window_qss` builds its chrome selectors from the fixed `CHROME`/`SYSTEM_FONT_STACK` values and reserves the dashboard-theme values for a handful of canvas-only selectors via dedicated `canvas_*` / `tile_*` format tokens; chrome that paints in Python (sidebar/tab-strip/inspector/picker icons via `monochrome_icon`, the Settings nav QSS) imports `CHROME` directly instead of reading the live theme. A dark preset (e.g. *Midnight Slate*) therefore recolors the tiles only and leaves the rail, tabs, inspector and dialogs untouched. (The HTML export, being pure canvas with no chrome, keeps the theme fonts/colors throughout.) The **element gap** is a transparent inset rendered *inside* each `GridTile` (`set_inset`, applied in `DashboardCanvas.place` and scaled by zoom) so adjacent cards always keep breathing room without moving any tile's footprint; it persists as a top-level `gap` in the v3 blob (default 0 = cards may touch). **The dashboard has two modes, toggled by the lock button on the tab strip (`_on_lock_toggled` → `_set_editing_locked`, persisted as the v3 `locked` bool).** **Build mode** (unlocked) = tiles move/resize, their ⚙/✕ buttons + right-click menu are shown, text/header double-click-edit works, and tile **contents are inert** (a chart click pushes no filter, the map left-drag moves the tile). **Use mode** (locked) = geometry is fixed, editing chrome is hidden, and **contents become interactive** (chart click → filter, list select → `featureAction`, map pan/identify/fly-to). The lock fans out `DashboardCanvas.set_locked` → `GridTile.set_locked`, which hides the editing chrome **and** calls `element.set_interactive(locked)` (the base no-op overridden by chart/pivot/list/category_selector/text/map to gate their handlers); the header is an ordinary tile gated by that same `set_locked`/`set_interactive` path. On load, `migrate_layout` defaults the mode from content via the pure `layout_util.default_locked` (a dashboard **with tiles opens in Use mode**, an empty one in Build mode), so older blobs (no `locked` key) open ready to use. A blank `new_dashboard()` and picking an element to add both drop to Build mode. **Cross-filter wiring is not on the rail** — it lives in a **per-tile right-click menu**: right-clicking a `GridTile` opens `Configure…` / `Connections…` / `Tile appearance…` / `Remove tile` (`GridTile.contextMenuEvent` → `configureRequested`/`connectionsRequested`/`styleRequested`/`closeRequested`). These three editors **do not open as modal dialogs** — they are hosted, one at a time, in a right-edge **inspector panel** (`side_panel.py → InspectorPanel`) that **overlays the right strip of the canvas** (an ArcGIS-Experience-Builder-style docked panel that never blocks the whole canvas; the left rail stays clear). The panel is a manually-positioned child of the central widget, raised above the page view and re-pinned in `DashboardWindow.resizeEvent`; it shows a title + ✕, the editor body, and a `Cancel`/`OK` footer. **All three preview live**: edits apply to the tile immediately, **OK** keeps them and **Cancel/✕** reverts to a snapshot taken on open; opening another editor implicitly commits the open one, and switching pages / removing the edited tile closes it (`InspectorPanel.open_editor`/`close_active`/`discard_if_subject`, wired from `_edit_tile_config`/`_edit_tile_style`/`_edit_tile_connections`). Each editor body is an embeddable `QWidget` extracted from the former dialog (the dialog classes remain as thin wrappers for standalone use / tests). `Configure…` hosts `add_element_dialog.py → ElementConfigForm` (`element=…` locks the type, prefills every row, exposes `managed_keys()`; `_edit_tile_config` replaces managed keys wholesale on a **debounced** `changed` signal — a cleared field drops its key — keeping `id`/`style`/`base_filter`, then `reconfigure()`s the tile). `Tile appearance…` hosts `appearance_dialog.py → AppearanceForm` (`mode="element"`, writing/clearing `config["style"]` live; a "Clear overrides" button drops the override). `Connections…` hosts `connections_dialog.py → ConnectionsForm` (`live=True`, each ticked edge wires immediately via `set_connected`; Cancel restores the page's `connections_to_dict` snapshot) editing *that one tile's* links from its own perspective — an outgoing "This filters →" section (its targets) and/or an incoming "← Filtered by" section (its sources), so dual-role tiles (chart, pivot) show both. The **header (brand banner)** is now an ordinary canvas tile, configured through the same per-tile `Configure…` inspector as any element (no global/per-page scope). The **grid resolution** dialog (`GridSettingsDialog`) opens from a **right-click on the empty canvas** (`DashboardCanvas.gridSettingsRequested` → `open_grid_settings`), since it's a layout setting rather than a frequent action. The live cross-filter indicator (a small accent dot + "Filters active: N") lives in the `QStatusBar`. The central widget is the rail beside a **content `QStackedWidget`** that swaps between the **Start screen** (`start_view.py → StartView`, index 0) and the page column (a `QTabBar` + a `QStackedWidget` of pages, index 1); `show_start()`/`show_dashboard()` switch them. Each page is a `DashboardPage` (`{id, title, view}`) wrapping a `PageView`. The tab bar **auto-hides when there is only one page** (`_update_tabbar_visibility`), so a single-page dashboard reads as just rail + canvas; it reappears once a second page is added. Tabs add (`+`/`Add page`), rename (double-click), delete (right-click; the last page can't be deleted), and reorder (drag). `Clear filter`/`Add element` and a tile's `Connections…` act on the **current** page; `Grid…` applies one **global** grid to every page. It owns the `DashboardBus`, creates elements via the registry factory, and handles **persistence**: the whole dashboard serializes to one **v3** dict via `_build_layout_dict()` — `{version:3, grid:{cols,rows}, gap, canvas:{w,h}, theme, window:{w,h}, locked, active_page, pages:[{id, title, connections, elements:[…incl per-tile grid+style]}]}` — the top-level `canvas` is the **export/print region** (the page size — a global one applied to every page); and `save_to_project()` writes it as JSON under project scope `QgisDashboard`/key `layout` (skipped when there are no pages, so an untouched project reopens to the Start screen). `_apply_layout_dict(data)` is the inverse, rebuilding every page/tile/theme from a migrated dict; legacy top-level (global) or per-page `header` keys are **materialized into header tiles** per page on load (`header_layout.materialize_header_tiles`). `load_from_project()` runs everything through the pure `migrate_layout()` helper, which upgrades the **v1** bare-list and **v2** single-page-with-top-level-connections blobs into v3 (collapsed into one "Page 1") and carries any legacy `header` keys through for that materialization (absent in newer blobs, where the header is just a tile). The **`canvas` region is optional** (absent in any pre-region blob): `_resolve_canvas_size()` then derives it from the content bounding box (rounded up) so existing exports keep their extent, falling back to the 1280×720 default when there are no tiles. When the `.qgz` has **no** dashboard, `load_from_project()` shows the **Start screen** instead of an empty page; the plugin's New-Project hook (`qgis_dashboard.py → _on_new`) does the same. The **Start screen** (`start_view.py → StartView`) is a themed wall of cards shown inside the canvas area — *New Dashboard* (`new_dashboard()`), *Open from file…* (`open_from_file()`), a *Continue current dashboard* card when one is loaded, and one **recent card** per recently saved/opened `.qdash` (branded preview + name/path/date, `openRecentRequested` → `open_file_path`). Recents live in **`QSettings`** via `recent_store.py → RecentStore` (capped at 8, deduped, auto-pruning missing files; pure `prune_missing`/`dedupe_insert` helpers). The portable **`.qdash`** file (`project_io.py`) is the same v3 dict + a `"_format"` marker, written/read by `write_layout_file`/`read_layout_file` and migrated on load exactly like the `.qgz` blob; opening one replaces the current dashboard (confirm-on-replace if it has tiles). Tiles still bind to layers by `layer_id`, so a `.qdash` restores fully on its own project and needs tile-rebinding on a different one. The header is now an **ordinary canvas tile**: `add_element("header", …)` goes through the normal `_add_element_to` tile path, not a separate dock. `closeEvent` emits `closed` so the toolbar action unticks.

3. **The page view (`page_view.py` → `PageView`)** is a thin `QWidget` container wrapping a private `_CanvasScroll` (a `QScrollArea` wrapping one `DashboardCanvas`). `_CanvasScroll` owns the page's **zoom/pan**: the canvas surface is the **export/print region** (the page) scaled by zoom; the scroll area is centered (`AlignCenter`) so a page smaller than the viewport sits framed, and overflows to scrollbars (or middle-mouse drag) when zoomed in. Range 0.1–4.0 via `set_zoom`/`zoom_in`/`zoom_out`; **`reset_zoom` fits the region to the viewport** (the pure, Qt-free `zoom_fit.fit_zoom(region, viewport, margin)` helper, unit-tested in `test/test_zoom_fit.py`) rather than snapping to 1.0 — so Reset Zoom always frames the exact rectangle that exports. Zoom is **view-only and never persisted**; the window reframes the current page on load/create via `_schedule_reframe()` (deferred to the next show when hidden). `PageView` delegates that zoom API to `_CanvasScroll`. The header is an ordinary canvas tile now, so the page no longer docks a banner; `PageView.export_pixmap` is just the canvas region render.

4. **The canvas (`dashboard_canvas.py` → `DashboardCanvas` + `GridTile`)** is the free drag/resize layout (one per page). Tiles are absolutely-positioned children remembering their placement in **cell units** (gx, gy, gw, gh); cell pixel size is `width/cols × height/rows`, so resizing/zooming or changing the grid rescales every tile. Drag a tile by its transparent top strip; resize via **8 handles** (4 corners + 4 edges — `_ResizeHandle`, geometry computed by the pure `_proposed_resize(edge, …)` helper). On release the tile **snaps to the grid**, is clamped in-bounds, and **reverts if it would overlap** another tile. Faint guides reveal the otherwise-invisible grid. Grid resolution is set in `settings_dialog.py → GridSettingsDialog`. The canvas carries the **export/print region** (`region_w`/`region_h`, `set_region`/`region_size`, default 1280×720) — the "page" the dashboard is laid out on: `paintEvent` frames it with a soft hairline and dims the off-page area with a faint scrim (suppressed while exporting via the `_exporting` flag), `sync_size()` grows the surface to `max(region, content)` so off-page tiles stay reachable, `first_free()` places new tiles within it, and **`export_pixmap()` renders exactly the region rect** (full-surface render, then crop) — so PNG/PDF are always the clean rectangle the user defined, with off-page tiles cropped and empty in-region space as margin.

5. **The theme (`theme.py` → `Theme`)** is the single source of truth for every color/font/metric. It carries a **body font** (`font_family`) and an optional **heading font** (`heading_font`, empty = reuse the body) so themes are real *pairings*: `heading_stack()` (heading → body → safe fallbacks) styles tile titles + the indicator value, `font_stack()` everything else. The global theme is edited in `appearance_dialog.py → AppearanceDialog` (a **preset gallery** combo, color pickers, a series-palette editor, and two `QFontComboBox`es — body + heading — listing all QGIS/Qt fonts); a tile may carry a partial override in `config["style"]`, merged via `Theme.merged_with`. **Preset themes live in `presets.py`** — a curated gallery of 12 coordinated palette+pairing looks (light + dark, drawn from `ui-ux-pro-max`). Each preset is a *partial* theme dict; `theme_for(name, base)` layers it over the current theme (keeping radius/sizes), and `match(theme)` reverse-maps a live theme back to its preset so the combo preselects (or shows `Custom`). Selecting one applies live; manual edits flip the combo to `Custom`. The window paints container chrome from `Theme.window_qss`; chart widgets read colors straight off the theme via each element's `_restyle()`.

6. **Elements (`elements/`)** are the dashboard tiles. `base.py → DashboardElement` (a `QFrame`) encodes the shared anatomy — title / body / description — a stable `id` (persisted in `config`, used by the connection graph), the **data-binding contract**, and appearance hooks:
   - `config["layer_id"]` — the bound `QgsVectorLayer` id (resolved lazily via `layer()`); `config["base_filter"]` — optional element-local expression.
   - `_combined_filter()` ANDs `base_filter` with `bus.combined_filter_for(self.id)`.
   - `iter_features()` yields features under the combined filter; `evaluate(expr)` runs an aggregate QgsExpression and exposes the combined filter as `@dashboard_filter`.
   - Class flags `is_filter_source` / `accepts_filter` declare an element's role. Subclasses implement `refresh()` (redraw) and optionally `_restyle()` (repaint custom views on theme change). Base wires `filtersChanged`/`layersChanged`/`themeChanged`/`filtersCleared` automatically.

### Source vs. target elements

| Element (`type_name`) | File | Role | Notes |
|---|---|---|---|
| `indicator` | `indicator.py` (+ `media.py`, `indicator_anim.py`) | target | Big value + optional reference/trend, via `evaluate()`. Configurable like a real card: an **icon** (PNG/JPG/SVG via `media.icon_pixmap`, placed left/right/above), an explicit **value text size**, and a value **animation** — `odometer` / `rolling` / `typewriter` / `fade` — played on change (`indicator_anim.IndicatorValue`; rolling is the custom-painted `_RollingNumber`). |
| `chart` | `chart.py` + `charts/painters.py` | source + target | One element, six QPainter chart types selected by `config["chart_type"]` — `bar`, `barh`, `line`, `area`, `pie`, `donut` — grouped by a category field with `count`/`sum`/`mean`. Clicking a bar/point/slice calls `set_filter`; clicking again clears. Pie/donut fold the long tail into an "Other" slice (not clickable). |
| `pivot` | `pivot.py` + `pivot_engine.py` | source + target | Cross-tab/matrix in a themed `QTableWidget`: rows = a row field, optional columns = a column field, cells aggregate a value field (`count`/`sum`/`mean`/`min`/`max`) with grand totals. The pandas-free `pivot_engine.compute_pivot` buckets `iter_features()` (testable on plain dicts). Clicking a data cell pushes `"row = r AND col = c"`; a row-header cell filters by row; a column header filters by column; clicking again clears. |
| `list` | `list_element.py` | target (featureAction source) | Feature table; selecting a row emits `featureAction` (it does not push a filter). |
| `map` | `map_element.py` (+ `map_filter.py`, `map_identify.py`) | source **+ fly-to target** (`accepts_filter=False`, `is_filter_source=True`) | A **live mirror of `iface.mapCanvas()`** whose behavior is **mode-dependent** (driven by the layout lock via `set_interactive`). In **Build mode** (unlocked) the hosting `GridTile`'s **full-tile drag overlay** owns the mouse like every other tile — a left-drag anywhere on the map moves the tile and a right-click opens the tile menu — while the map itself is inert (mirrors the QGIS extent, pushes nothing). In **Use mode** (locked) a `_PanIdentifyTool` (a `QgsMapToolPan` subclass) is installed on the canvas: a left-**drag** pans the tile's own canvas (native `QgsMapToolPan`, so no middle-mouse needed) and a left-**click** (movement under `CLICK_THRESHOLD` px, decided on release) **identifies** the bound layer's feature under the cursor in a small themed `IdentifyPopup` (pixel→map coords → `map_identify.search_rect` tolerance box → first hit → `map_identify.feature_summary`; the pure helpers are unit-tested). As a **fly-to target** it reacts to `filtersChanged`: `_fly_to_filtered` computes `combined_filter_for(self.id)` (AND of its **wired** sources only), queries the bound layer, and flies+flashes to the union of matching features' bounding boxes (one feature → its bbox; cleared → re-mirror the QGIS extent), deduped by `_last_fly_expr` so the map's own extent pushes never re-trigger a fly. As a **spatial source** it pushes a debounced (`200 ms`) extent filter from **its own** `canvas.extent()` so connected tiles re-query to the visible frame: `map_filter.extent_filter_expression` (pure, CRS-safe — `transform($geometry, layer_property(@layer,'crs'), <map CRS>)` intersected with the extent WKT, `coalesce(…, true)` so non-spatial targets are a no-op). Pushes are gated on `isVisible()` **and** Use mode so a Build-mode/off-page map never filters; `_sync_extent` mirrors the iface extent **only in Build mode** so a Use-mode pan isn't overwritten. The `extent_filter_enabled` config flag (default on) pauses the push without unwiring. `featureAction` (a list row pick) still zooms/flashes. In the HTML export the map is an interactive Leaflet map (online basemap — the project's XYZ layer if present, else OpenStreetMap — with the dashboard's layers drawn as clickable WGS84 vector features and an all-fields identify popup); it does not cross-filter, and a static canvas snapshot is embedded only as a fallback when Leaflet can't initialize. The basemap needs network; the rest of the file stays offline. |
| `category_selector` | `category_selector.py` | source only (`accepts_filter=False`) | Dropdown of unique values; the cleanest source. A pure source must not react to filters — set `accepts_filter = False` (don't filter itself down to one value). |
| `text` | `text_element.py` | neither (presentational) | A free-text / heading container (no layer, no cross-filter). Hides the base title/description chrome and fills the tile with one themed `QLabel`; **double-click the tile** to edit the text (multi-line prompt). `config`: `text`, `align` (left/center/right), `heading` (bool → large+bold). |
| `image` | `image_element.py` | neither (presentational, `full_bleed`) | Shows an image file scaled to the tile — PNG/JPG/BMP/WEBP via `QPixmap`, SVG via `QSvgRenderer`, animated GIF via `QMovie`. `config`: `path` (a **file-path reference**, not embedded — the file must stay reachable), `fit` (`contain` keep-aspect / `stretch`). |
| `header` | `header.py` (+ `header_layout.py`) | neither (presentational) | A **brand banner that is now an ordinary canvas tile** (`GridTile`): free drag/resize/snap, region-cropped on export, configured via the inspector and gated by the Build/Use lock like any element. Carries a styled title (`font_family` from any installed QGIS/Qt font via the dialog's `QFontComboBox`, `font_size`, `align`) and one logo image in an anchored slot (`logo_slot`: left/right/above/below, `logo_path`, `logo_size`). New header tiles get a banner-shaped default (full region width × `HEADER_BAND_H`). Because height is **tile geometry, not config**, the header's `Configure…` form adds a non-config **"Banner height" (px)** row (only when editing a live tile) seeded from the tile and written straight to its pixel height via `GridTile.set_height_px` — a numeric stand-in for dragging the south resize handle (exact, no grid-snap/`MIN_TILE` floor). Instead of reverting when tiles sit below it, the banner **pushes the whole stack below it** as one accordion: every other tile whose top is at/below the banner's bottom is shifted by the height delta (down as it grows, up as it shrinks), which is overlap-safe (any same-column tile below already starts at/after the banner's bottom). `_edit_tile_config` applies it live and Cancel restores the snapshot height (reversing the push). It is excluded from `ElementConfigForm.managed_keys()`/`result_config()` so it never leaks into the element config. Legacy **docked** headers (the old top-level/per-page `header` blob keys, with `anchor`/`thickness`) are migrated into header tiles on load by `header_layout.materialize_header_tiles` (+ `header_tile_placement`), which shifts existing tiles out of the band and grows the region. The pure `header_layout` helpers (`inner_box_direction`, `header_tile_placement`, `materialize_header_tiles`, `resolve_header`) are Qt-free. |

The factory and labels live in `elements/__init__.py` (`ELEMENT_TYPES`, `ELEMENT_LABELS`, `create_element`). `create_element` also **migrates the legacy `serial_chart`/`pie_chart` element types onto `chart`** (with the right `chart_type`), so dashboards saved before the registry still load; they auto-upgrade to `__type__:"chart"` on the next save. The per-type configuration UI is `add_element_dialog.py → AddElementDialog`, which builds dynamic rows per selected type and returns `(type_name, config)`.

**Chart types are a declarative registry with data shapes.** `elements/chart_specs.py → CHART_SPECS` maps each `chart_type` to its painter key, label, group, **data `shape`**, whether it takes a statistic/value field, "Other"-folding, and donut inner-radius (plus the Qt-free helpers `fold_categories` / `filter_literal` / `shape_of`). There are **23 chart types** across the shapes: `category` (bar, barh, lollipop, lollipop_h, dot, radial_bar, radar, line, step, spline, area, waterfall, pie, donut, rose, funnel, treemap), `series` (grouped_bar, stacked_bar), `xy`/`xyz` (scatter, bubble), `bins` (histogram), `ohlc` (candlestick). `elements/charts/painters.py → PAINTERS` maps painter keys to `_ChartPainter` subclasses (each emits `categoryClicked`; category painters draw uniform `[(category, value)]` data, the others draw their shape's payload). **The `shape` determines the data contract:** `ChartElement._produce()` dispatches on it — `category` uses `_aggregate()` (the original `[(category, value)]`), while `series`/`xy`/`xyz`/`bins`/`ohlc` call the **Qt-free producers in `elements/chart_data.py`** (`aggregate_series`, `collect_points`, `histogram_bins`, `aggregate_ohlc`, plus the `squarify` treemap-geometry helper — all unit-tested in `test/test_chart_data.py`). Cross-filtering is shape-aware: `category`/`series`/`ohlc` clicks push a category-equality filter, **histogram** clicks push a numeric **range** filter (`"field" >= lo AND "field" < hi`), and scatter/bubble are display-only. `ChartElement.reconfigure()` **swaps the painter widget** when the chart type's painter class changes (the painter is chosen once in `__init__`, so without this every chart would stay a bar chart; it uses an **exact** `type(...)` check because `area`/`step`/`spline` subclass `line`). The HTML export mirrors all of this in `export/assets/runtime.js` (a parallel `CHART_SPECS` with `shape`, ports of the producers + `squarify`, and one SVG painter per type), so exports match the desktop; the runtime's selection-predicate model gained a `{field, lo, hi}` **range** predicate for histogram. **Adding a category-shaped chart type = one painter class + one `CHART_SPECS` row** (no change to `ChartElement` or the dialog); a new *shape* additionally needs a producer in `chart_data.py`, a branch in `ChartElement._produce()`, field rows in `ElementConfigForm._add_chart_rows()`, and the matching `runtime.js` aggregator + painter. The config form's chart field rows are **chart-type-reactive**: selecting a type rebuilds the rows its shape needs (e.g. scatter shows X/Y, histogram shows value+bins), preserving field selections across the switch.

### Plugin entry point

`__init__.py → classFactory(iface)` returns `qgis_dashboard.py → qgisdashboard`. That class keeps the Plugin Builder conventions (`tr()`, i18n loading, an `actions` list) but instead of opening a modal dialog it:
- registers one **checkable** toolbar + Plugins-menu action that shows/hides the standalone window;
- lazily creates the `DashboardWindow` on first toggle (`_ensure_window`) and unticks the action when the window's `closed` signal fires;
- connects `QgsProject.writeProject`, `iface.projectRead`, `iface.newProjectCreated` to save / load / clear the dashboard.

**Icon loading is intentionally resilient:** `from .resources import *` is wrapped in `try/except ImportError`. If `resources.py` was never compiled, the icon loads from the filesystem (`os.path.join(plugin_dir, 'icon.png')`) instead of the Qt resource path. This is what lets the plugin install without a build step — do not reintroduce a hard `from .resources import *`.

### Interactive HTML export (`export/`)

The Settings hub ("Export to HTML…", wired via `window.export_to_html` → `export_dialog.prompt_and_export`) writes the whole dashboard to a **single self-contained `index.html`** that opens offline by double-click, with **live cross-filtering** reproduced client-side. There is **no framework, no Node, no server** — that is a hard requirement driven by the `file://` protocol (it blocks `fetch`, ES-module `import`, and Web Workers), so everything (CSS, JS, data) is inlined, data is embedded in a `<script type="application/json">` block (read, never fetched), and there is no charting library; the only map dependency is Leaflet (vendored and inlined, so the file stays self-contained — only its basemap tiles stream from the network).

- **`export/serialize.py`, `export/theme_css.py`, `export/html_builder.py`** are **pure** (no QGIS/Qt) and unit-tested in `test/test_html_export.py`: they assemble the export-model dict, turn `Theme.to_dict()` into `:root` CSS variables (web mirror of `Theme.window_qss`), and inline the runtime + escaped JSON into the final HTML.
- **`export/data_collect.py`, `export/map_snapshot.py`, `export/html_export.py`** are the QGIS-touching layer: per-layer rows (all fields, deduped per layer), each tile's static `base_filter` evaluated server-side into a list of passing feature **indices** (so the browser needs no QgsExpression parser — it intersects index sets), image files base64-embedded, each referenced layer's geometry reprojected to WGS84 and embedded as a per-feature GeoJSON array (`export/geometry_collect.py`, index-aligned with its attribute rows) so the map tile becomes an interactive Leaflet map (`export/basemap.py` resolves the basemap; vendored, inlined Leaflet 1.9.4; online basemap tiles; the base64 canvas snapshot from `map_snapshot.py` kept only as the no-Leaflet fallback), and a per-layer size guard (`oversize_layers`, geometry-aware via `export/size_estimate.py`) feeding the Proceed/Skip/Cancel warning.
- **`export/assets/runtime.js` + `runtime.css`** are the **vanilla-JS browser runtime**, shipped as real asset files (not Python string blobs) and inlined at export. `runtime.js` ports `bus.combined_filter_for` + each source's click logic (sources push `(field, value)` equality predicates; targets AND them over their own rows), `chart._aggregate` + `fold_categories`, and `pivot_engine.compute_pivot`; charts are hand-drawn inline SVG (bar/barh/line/area/pie/donut). Indicators evaluate the common aggregate-expression forms client-side and fall back to a server-computed `indicator_value` for anything unsupported; the indicator's **icon** (base64 `icon_uri`, embedded by `html_export` via `image_data_uri`), **value text size**, and **value animation** (`odometer`/`rolling`/`typewriter`/`fade`, easing mirrors `indicator_anim`'s `OutCubic`; rolling falls back to a count-up under `file://`) are reproduced in `runtime.js`/`runtime.css`. The **map** is an interactive Leaflet map (pan/zoom + click-to-identify all fields) but does not cross-filter — that remains desktop-only — and its online basemap tiles require a network connection.
- Registered in **both** `pb_tool.cfg` (`python_files` += `export_dialog.py`; `extra_dirs` += `export`) and `Makefile` (`PY_FILES`/`SOURCES` += `export_dialog.py`; `EXTRA_DIRS` += `export`). The `export/` package ships via `extra_dirs` (like `elements/`), so its `assets/` go along.

Design spec: `docs/superpowers/specs/2026-06-16-html-export-design.md`.

This is a QGIS/PyQt5 plugin; there is no app build. "Running" means loading it inside QGIS. The QGIS Python environment must be on `PYTHONPATH` for tests and imports (`qgis.core`, `qgis.gui`, `qgis.PyQt`).

## Conventions specific to this codebase

- **Stay inside the Plugin Builder layout.** When you add a Python module or an `extra_dir`, register it in **both** `pb_tool.cfg` (`python_files` / `extra_dirs`) and `Makefile` (`PY_FILES` / `SOURCES` / `EXTRA_DIRS`), or it won't ship in the packaged zip.
- **Relative imports within the package** (`from .bus import …`, `from .elements import …`). The QGIS plugin loader imports the folder as a package; absolute imports break on install.
- **Qt comes through `qgis.PyQt`**, never bare `PyQt5`. **Do not use `qgis.PyQt.QtChart`** — the QtChart module is optional and is absent from many QGIS/PyQt builds (it raised `ModuleNotFoundError` on install). Charts are drawn with plain `QPainter` in a custom `QWidget` (see `elements/charts/painters.py`); follow that pattern for new visualizations.
- **Qt5/Qt6 dual compatibility (QGIS 3.22 – 4.99).** QGIS 4.0 runs on Qt6/PyQt6, where Qt5-era patterns raise `AttributeError` at runtime. The plugin targets both, so **always use fully-scoped enums** — `Qt.AlignmentFlag.AlignLeft`, `QPainter.RenderHint.Antialiasing`, `QImage.Format.Format_ARGB32`, `QDialogButtonBox.StandardButton.Ok`, etc. — never the unscoped `Qt.AlignLeft` form (works on Qt5 only). Use `.exec()` not `.exec_()` (removed in PyQt6). Scoped enums and `.exec()` are valid in **both** PyQt5 and PyQt6. `metadata.txt` must keep `qgisMaximumVersion=4.99` (the default `<major>.99` would otherwise exclude 4.x). Mouse-event `pos()`/`globalPos()` are kept as-is (deprecated but bound in PyQt6; the Qt6-only `position()` would break Qt5).
- **Never mutate shared project layers** to apply a dashboard filter (no `setSubsetString` on project layers) — filtering is done per-element through `QgsFeatureRequest` so other plugins/views are unaffected.
- **Adding a new element type** (the most common extension):
  1. Create `elements/<type>.py` with a `DashboardElement` subclass, set `type_name`, set the `is_filter_source` / `accepts_filter` class flags, implement `refresh()`. To make it a *source*, call `self.bus.set_filter(self.id, expr_or_None)`; for a pure source set `accepts_filter = False` (see `category_selector.py`). For a theme-aware custom-painted view, override `_restyle()` and call `self.apply_theme()` in `__init__`.
  2. Register the class in `elements/__init__.py` (`ELEMENT_TYPES`) and add a label to `ELEMENT_LABELS`. Add a matching `el_<type_name>` stroke glyph to `icons.ICONS` so the Add-element picker shows it (it derives the icon key from the type name automatically; the label is the hover tooltip).
  3. Add its config rows in `AddElementDialog._rebuild` and read them in `result_config` — this is the tile's `Configure…` (edit) form. The *add* picker takes no per-type input, so the tile is created with default config and configured here.
  4. Persistence is automatic via `base.to_dict()` (whole `config` + `__type__` + `id`); the window adds `grid` placement. Ensure config values are JSON-serializable.
- **Theme everything through `Theme`.** New visualizations must read colors/fonts from `effective_theme()` (global merged with the tile's `config["style"]`), never hardcode — so the Appearance dialog and per-tile overrides keep working.
- **Borders are always soft hairlines — never dark/heavy black outlines.** Use `theme.border` (the `#e2e6ec` hairline, the same one the left rail uses: `border-right:1px solid {border}`) for any separator, edge, or divider. Do not introduce dark/near-black outlines (`#000`, `#111`, thick borders) for chrome — they read as heavy and unprofessional. If a platform widget paints its own dark line (e.g. `QTabBar`'s native base line), disable it (`setDrawBase(False)`) and supply the theme hairline via QSS instead. Prefer subtle backgrounds (`brand_soft`, `selection`) over outlines for emphasis.

## Known limitations (deliberate, carried over)

- **No free-form overlap / z-ordering.** Tiles snap to the grid and may not overlap; placement reverts on collision. Resolution is set by the grid cols/rows.
- **Aggregates run on the UI thread.** For million-feature layers, move `_aggregate()` / `evaluate()` into a `QgsTask`.
- **QGIS aggregate functions ignore the feature-request filter**, so a bare `count(1)` in an indicator does not auto-fold the dashboard filter. `base.evaluate()` exposes the combined filter as the `@dashboard_filter` variable; use the aggregate `filter:=` argument to opt in. Charts/lists, which iterate `iter_features()`, *are* filter-aware.
- **Desktop only** — no web publishing; that is structural to any QGIS plugin.

# Summarizer — Capability Research & Analysis

> **Repo:** https://github.com/jeandsonmarques/Summarizer
> **Author:** Jeandson Marques · **License:** GPL-3.0-or-later · **Version analyzed:** 0.5.6 ("First stable QGIS 3.x release")
> **Type:** QGIS 3.x desktop plugin (QGIS 3.34 → 3.99; QGIS 4.x not declared)
> **Scale:** ~57,600 lines of Python across 119 files, 11 subpackages
> **Researched:** 2026-06-15 via deep code read (not just README) of a shallow clone.

This document maps **what Summarizer actually does and how it is built**, so it can be imitated and improved in your own `QGIS Dashboard` plugin. The final section is a direct, actionable comparison against your current project.

---

## 1. What Summarizer Is

A self-described *"QGIS plugin for layer summaries, charts, dashboards, and report-ready analytical outputs."* In practice it is far more ambitious than the tagline — it is essentially a **Power-BI / Tableau-style business-intelligence environment embedded inside QGIS**, working off the project's vector layers *and* external data sources (Excel, CSV, PostgreSQL/PostGIS, SQL Server).

It bundles six major capabilities under one window:

| # | Capability | What it gives the user |
|---|-----------|------------------------|
| 1 | **Layer/attribute summaries** | Group-by + descriptive statistics (mean, median, std, percentiles) over any vector layer field. |
| 2 | **Pivot tables** | Full rows/columns/values/filters cross-tabs with subtotals, grand totals, % of row/col/total. |
| 3 | **Charts & dashboards** | 18 chart types rendered with custom `QPainter`, arranged on a multi-page free-form canvas with cross-filtering. |
| 4 | **Data model view** | A Power-BI-style entity-relationship canvas (table cards + relationship lines) that drives cross-tile filtering. |
| 5 | **Database explorer** | Browse PostgreSQL/PostGIS (and partial SQL Server) schemas/tables/views, with secure credential storage. |
| 6 | **Reports / export / presentation** | Export-ready tables, Excel/CSV/JSON/PDF output, and a side-by-side "presentation" map mode. |

Core workflows run **locally** inside QGIS with no external services. The UI is trilingual (Portuguese / English / Spanish) with live language switching.

## 2. High-Level Architecture

### Entry point & shell
- **`__init__.py → classFactory(iface)`** returns the main `Summarizer` class (defined in `data_summarizer.py`). Before that it does two notable things:
  - **`_prioritize_qgis_binary_packages()`** — detects other plugins that prepend a `vendor_py` folder containing their own `numpy`/`pandas` to `sys.path`, and demotes those paths so QGIS's own numpy/pandas win. This prevents the classic "numpy binary incompatibility" crash when multiple plugins vendor scientific libraries. **Worth stealing if you ever depend on pandas.**
  - Registers custom **Inter** UI fonts (bundled TTFs).
- **`initGui()`** registers a toolbar action + a Plugins-menu entry, and injects a **custom Summarizer node into the QGIS Browser** (`browser_integration.py`) so saved DB connections appear in the native browser tree.
- The UI is a **modal `QDialog`** (1200×800, programmatically built in `ui_main_dialog.py` — no `.ui` file), **not a dock**. (Your project deliberately chose a standalone top-level window instead — a reasonable divergence.)

### Navigation model
- **`layout_nav.py → SidebarController`** drives a compact **icon-only vertical sidebar** with an animated active-indicator bar. Pages live in a `QStackedWidget`:
  - `resumo` → Summary/Results · `model` → Data Model · `database` → Database Explorer · `integracao` → Integration/Connections.
- **Deferred page building**: heavy pages (Model, Integration, DB) are built on first access via a `_deferred_page_build_queue` + `QTimer.singleShot(~350ms)`, keeping startup fast.
- The database button shows a colored **status dot** (green/yellow/red/gray) for connection state and auto-hides when no DB connections exist.

### Cross-cutting infrastructure
- **`palette.py`** — single source of truth for `COLORS`, `DARK_COLORS`, `TYPOGRAPHY`, spacing; stylesheets are generated by template substitution. (Conceptually the same role as your `theme.py`, but token-dict based rather than a `Theme` dataclass.)
- **Dual i18n**: compiled Qt `.qm` translations for static strings **plus** a JSON runtime layer (`utils/i18n_runtime.py`, ~90 KB) — `runtime_{en,es}.json` machine translations + `runtime_overrides_{en,es}.json` human fixes, with a Portuguese fallback chain. Enables **live language switching without restart**.
- **Security**: `utils/security_utils.py` stores DB credentials through the **QGIS AuthManager** (`QgsAuthMethodConfig("Basic")`), persisting only an `authcfg` UUID token — **never plaintext passwords** in QSettings. `secure_connection_payload()` stores, `reveal_connection_payload()` lazily decrypts at connect time. `logging_utils.sanitize_log_message()` redacts passwords/tokens/connection strings from the QGIS log.
- **Signal-based loose coupling** + module-level session caches for connected DB params/tables.

## 3. Dashboard Subsystem (the part most relevant to you)

### Data model (`dashboard_models.py`, ~70 KB, dataclass-based)
A clean, type-safe, serializable hierarchy:

```
DashboardProject (version 2, project_id, created_at/updated_at, edit_mode)
└── DashboardPage[]  (page_id, title, zoom 0.6–2.0, items, filters)
    └── DashboardChartItem[]  (item_id, payload, visual_state, binding, layout, title…)
        ├── DashboardChartBinding   → role-based field mapping
        ├── ChartVisualState        → ~67 styling properties
        └── DashboardItemLayout     → dual grid + pixel positioning
```

- **Field binding is role-based** (`SLOT_DEFINITIONS`): `X_AXIS`, `Y_AXIS`, `VALUES`, `LEGEND`, `FILTERS`, `TOOLTIP`, `SIZE`. Each slot can hold typed `FieldBindingItem`s with an aggregation (`sum/count/avg/min/max/unique_count/none`). Adding a new chart type = adding a `SLOT_DEFINITIONS` entry — **no rendering-code change needed to declare it.**
- **Dual layout system** — items store *both* pixel (`x,y,w,h`) and grid (`row,col,row_span,col_span`) coordinates, auto-converting between them (`x = 24 + col*294`, etc.). Gives precise control *and* responsive grid snapping.

### Layout canvas (`dashboard_canvas.py`, ~79 KB)
- Free-form positioned child widgets (not `QGraphicsScene`), with **zoom 0.35–3.0×**, pan (middle-mouse / spacebar), 8 px snap grid with guides, z-ordering, and an **8-handle resize/selection overlay** (`_DashboardConnectorOverlay`).

### Supported tiles (18 types)
`bar`, `barh`, `column_clustered`, `column_stacked`, `bar100_stacked`, `line`, `area`, `pie`, `donut`, `funnel`, `treemap`, `waterfall`, `combo`, `scatter` (with size channel → bubble), `matrix` (pivot), `table`, `slicer` (filter control), `card`, `kpi`, `gauge`.

### Cross-filtering (explicit relation graph)
- **`DashboardChartRelation`**: `source_chart_id → target_chart_id`, with `source_field`/`target_field`, `interaction_mode` (`filter`/`none`), and **`direction`** (`both` / source→target / target→source) — each relation toggleable.
- **`DashboardVisualLink`** draws the wire on the canvas with **orthogonal routing + lane management** so multiple links don't overlap.
- **`ModelInteractionManager`** propagates a selection in one tile to connected tiles along relation paths, supporting bidirectional flow. (Your project's `DashboardBus` does the same job via a connection graph + signals — same philosophy, different plumbing.)

### Persistence
- Saved as **standalone JSON files** (`.pbsdash` extension) under `~/Documents/Summarizer/Dashboards/`, with `to_dict()/from_dict()` on every model class, version field (v2), ISO timestamps, UUID fallback for missing IDs, and **field normalization/clamping on load**. Recent-projects list (8) in QSettings.
- *(Contrast: your project persists the dashboard layout JSON inside the `.qgz` project file under scope `QgisDashboard`/key `layout`. Summarizer keeps dashboards as portable external files — a different, shareable model.)*

### Styling
- 4 built-in presets (`clean`, `executive`, `focus`, `graphite`), per-tile overrides for colors/typography/borders/number-formatting/grid, **copy-paste styles between tiles** (clipboard global), dark/light aware, with a live-preview `VisualFormatPanel`.

## 4. Charts & Reports (`report_view/`)

- **Rendering tech: 100% custom `QPainter`** in `chart_factory.py` (the single biggest file at **275 KB**), via `ReportChartWidget(QWidget)` → `paintEvent` → `_dispatch_chart_draw()` → per-type `_draw_*` methods. **No QtCharts, no matplotlib, no web view.** *(This independently validates your project's exact decision to avoid `qgis.PyQt.QtChart` and draw with `QPainter`.)*
- **Animation system** (`charts/chart_animation.py`): tiered durations (hover 185 ms, selection, filter 300 ms, data 320 ms, entry 360 ms, type-change 390 ms), intensity levels (normal/reduced/off), easing curves (OutCubic/OutQuart/InOutCubic), staggered per-item timing, value count-up, and frame-snapshot crossfades between visual states.
- **Smart defaults** (`_choose_chart_type`): date/datetime → line; 2–5 categories + aggregate → pie; else bar.
- **Performance guards**: `MAX_RENDER_ITEMS = 160`, signature-based change detection (`_payload_signature`) to skip redundant repaints, color caches, matrix-view scrollbar virtualization.
- **Accessibility**: alt-text per chart, keyboard navigation, focus handling.
- **Reports** (`report_view/reports/`): a "report" = export-ready bundle (`headers`, `rows`, helper text, value label) with locale-aware number formatting and HTML/CSS templates for preview.

## 5. Pivot Tables (`pivot/`, `pivot_view/`, `report_view/pivot/`)

Three deliberately separate layers (not legacy duplication):

- **`pivot/`** — pure data/formatting/export helpers (pandas-oriented, UI-free).
- **`pivot_view/`** — the QGIS UI: `pivot_table_widget.py` (132 KB), drag-drop field areas (rows/cols/filters/values via a custom MIME type), `QStandardItemModel` + `QSortFilterProxyModel` rendering, themed alternating rows, toolbar, undo/redo state controller, settings dialog.
- **`report_view/pivot/`** — a **layer-backed engine** (`pivot_engine.py`) that iterates features via `QgsFeatureRequest` and aggregates into custom `PivotBucket` accumulators (using Python's `statistics` module for median/variance/stddev) — used when working directly off QGIS layers rather than in-memory DataFrames.

Capabilities: 9 aggregations (`count/sum/average/min/max/median/variance/stddev/unique`), subtotals + grand totals, **% of total / row / column**, configurable null inclusion, and **every cell tracks `feature_ids`** → click a cell to select/zoom those features on the map (`pivot_selection_bridge.py`). **Geometry length/area are first-class aggregatable "fields."** A `PivotValidator` rejects invalid combos early (e.g., median on a text field). Export to CSV (pandas), static Excel (openpyxl), or **native Excel pivot tables via win32com COM automation** (with graceful fallback when Excel isn't installed).

## 6. Data Model View (`model_view/`, `model_tab.py` 211 KB)

A **Power-BI-style relationship diagram**, the most architecturally distinctive subsystem:

- Built on **`QGraphicsScene`/`QGraphicsView`** (unlike the dashboard canvas's child-widget approach). Table cards (`TableCardItem` = QGraphicsRectItem with header + field rows + drop shadow), fields (`FieldItem` with connection-port dots, PK=yellow / FK=blue markers), relationships (`RelationshipItem` = QGraphicsPathItem, curved/orthogonal/straight, with `1`/`*` cardinality glyphs and direction arrows).
- **Drag a field port to another field** to create a relationship (dashed live preview, validated on drop). Relationships carry **cardinality (1:1/1:\*/\*:1/\*:\*) AND a separate direction (forward/backward/both)**.
- Tables come from **QGIS vector layers *and* imported pandas DataFrames** (Excel/CSV imports become virtual tables). **Heuristic PK detection** (`primaryKeyAttributes()` or `id`/`*_id` naming).
- **Virtual field previews**: related cards show, in muted style, which cross-table fields will be available for filtering given the relationship direction + selected fields.
- Relationships **drive cross-chart filtering** (filters flow along relationship paths) and can build **unified layers** via `_run_simple_left_join()` into memory layers.
- Auto-layout for new tables, deduplication by a 6-tuple signature, undo/redo (80 snapshots), theme-aware rendering. Persisted to QSettings (`Summarizer/model/layout`) + project `source_meta`.

## 7. Data Sources, Database Explorer, Export, Presentation

- **Integration panel** (`integration_panel.py`, 123 KB): a hub of connector cards (Excel, CSV, PostgreSQL quick-connect, SQL Server, saved DB connections, QGIS layers) in a **responsive grid** (1–4 columns by width), plus a recents list. Imported datasets live as `Dict[str, pd.DataFrame]`.
- **Database explorer** (`database_explorer/`): PostgreSQL/PostGIS primary (direct `QSqlDatabase`/QPSQL queries against `information_schema`, PostGIS detection via `geometry_columns`, classifies BASE TABLE / VIEW / MATERIALIZED VIEW / PARTITIONED), SQL Server partial; metadata loads on a **`QThread` worker** to keep the UI responsive; expandable schema cards with lazy row loading.
- **Summary engine** (`data_summarizer.py` + `summary_view/`): pandas-first. `build_dataframe_summary` / `calculate_advanced_summary` produce basic stats + percentiles (P25/50/75/90/95 via numpy) + grouped aggregations, then convert back to QGIS types for materialization. `is_meaningful_value()` filters nulls.
- **Export manager** (`export_manager.py`): **Excel** (multi-sheet via pandas+openpyxl), **CSV**, **JSON**, **PDF** (custom `QPdfWriter`+`QPainter` layout — header, metadata, stats/percentile tables, top-10 groups, A4 auto-pagination). Layer materialization back to QGIS / GeoPackage via `QgsVectorFileWriter`.
- **Presentation mode** (`presentation/`): a checkable "Apresentar" button opens a **secondary `QgsMapCanvas` side-by-side** with the dashboard, synced to the main canvas, with pan/zoom map tools.

## 8. Engineering Quality Signals

- **Tests**: `tests/smoke/` (import/package/metadata/release-file integrity) + `tests/unit/` (~50 files covering pivot calc, dashboard binding/visual state, model persistence/relations, chart models/utils, report export, security utils). `conftest.py` puts plugin dir on `sys.path` and **skips QGIS-dependent tests when QGIS is absent**, so pure-Python logic is testable in plain CI.
- **CI**: `.github/workflows/python-package-conda.yml` — Python 3.10, `ruff check`, `compileall`, `pytest --cov` (no enforced threshold).
- **Release discipline**: dedicated `ROADMAP.md`, `RELEASE.md`, `STABILIZATION_PLAN.md`; `main` protected/stabilized, work on `develop`, `feature/*`, `hotfix/*`, `release/*`; ZIP packaged with a clean `Summarizer/` root; changelog shows iterative cleanup of Flake8 findings to pass `plugins.qgis.org` validation.
- **Defensive coding**: pervasive try/except with sanitized logging, deferred/worker-thread initialization, `__`-prefixed internal columns, dataclass normalization on every load.

### Notable strengths worth imitating
1. **`QPainter`-only charts** — zero charting deps, full control, install-anywhere (same as your approach — strong validation).
2. **Role-based binding + `SLOT_DEFINITIONS`** — declaratively add chart types.
3. **Dual grid+pixel layout** — precise *and* responsive.
4. **Relation graph with direction ≠ cardinality** — asymmetric/bidirectional cross-filtering.
5. **AuthManager credential handling + log sanitization** — security done right.
6. **Vendor-path numpy/pandas guard** — survives a messy multi-plugin ecosystem.
7. **Dual (compiled + JSON-runtime) i18n with live switching**.
8. **Power-BI data-model canvas** with drag-to-link and virtual-field previews.
9. **Feature-ID tracking in every pivot cell** → seamless map drill-down.

### Weaknesses / gaps (your opportunities to improve)
1. **Heavy pandas/numpy dependency** — bigger install, the vendor-path crash class, and a hard runtime requirement. Your project avoids this entirely.
2. **Two parallel rendering paradigms** (dashboard = child widgets, model = QGraphicsScene) and **three pivot module trees** — real cognitive overhead; formatting logic is fragmented across `pivot_formatting.py`, `pivot_formatters.py`, and inline renderer code.
3. **Giant files** — 275 KB / 211 KB / 132 KB single modules violate "small, focused files" and are hard to maintain/test.
4. **No schema migration logic** — a `version` field exists but load just deserializes as-is; v1→v2 upgrades aren't handled.
5. **No real cross-tile data binding** — relations only push *filters*; tiles are still independently rendered (no shared/joined dataset across tiles beyond the join-to-memory-layer path).
6. **Native Excel pivots need win32com + local Excel** (Windows-only, fragile) — fallback output differs from the "real" result.
7. **Aggregates on the UI thread** for summaries/pivots (only DB metadata uses a worker) — large layers can freeze the UI.
8. **QGIS 4.x not declared**; modal-dialog UX blocks interaction with the QGIS map while open (your standalone-window choice is better here).
9. **i18n primary language is Portuguese** with English as a translation layer — origin strings are PT.

## 9. Direct Comparison vs. Your `QGIS Dashboard` Project

Your project (per `CLAUDE.md`) is a Plugin-Builder-shaped dashboard plugin: a **standalone top-level window**, a **`DashboardBus`** signal/connection graph, a **free drag/resize grid canvas with snap+collision-revert**, a **`Theme`** single-source-of-truth, and a small set of `QPainter`-drawn element types (`indicator`, `serial_chart`, `pie_chart`, `list`, `map`, `category_selector`), with layout saved inside the `.qgz`.

| Dimension | Your QGIS Dashboard | Summarizer | Takeaway |
|---|---|---|---|
| **Chart rendering** | `QPainter` custom widgets | `QPainter` custom widgets | ✅ Same approach — you're on the right track. |
| **Chart variety** | ~2 chart tiles (serial/pie) + indicator/list/map/selector | 18 chart types | **Biggest gap.** Adopt their `SLOT_DEFINITIONS` pattern to scale up cheaply. |
| **Layout** | Grid snap, collision revert, cell-unit tiles | Dual grid+pixel, zoom/pan, 8-handle resize | Add zoom/pan + free pixel mode + resize handles. |
| **Cross-filter** | `DashboardBus` connection graph + signals | Relation graph w/ direction + visual wired links | Add **direction** (not just on/off) + on-canvas link wires. |
| **Multi-page** | Single canvas | Multi-page projects | Add pages. |
| **Pivot/matrix tile** | none | Full pivot engine + matrix tile | High-value addition; consider the layer-backed (non-pandas) engine pattern. |
| **Data model view** | none | Power-BI ERD canvas driving filters | Ambitious differentiator; QGraphicsScene-based. |
| **Data sources** | project vector layers | layers + Excel/CSV/PostgreSQL/SQL Server | Optional DB/file ingestion. |
| **Persistence** | inside `.qgz` (`QgisDashboard/layout`) | portable external `.pbsdash` JSON | Consider **both** — in-project *and* exportable. |
| **Theming** | `Theme` dataclass + per-tile `style` override | palette tokens + 4 presets + copy/paste styles | Add **style presets** + **copy-paste between tiles**. |
| **Dependencies** | none beyond QGIS/PyQt (deliberate) | pandas + numpy + openpyxl + win32com | ✅ Your lean approach is a genuine advantage — keep it. |
| **i18n** | (not emphasized) | dual compiled+runtime, live switch, PT/EN/ES | Optional; their runtime-JSON pattern is reusable. |
| **Security** | per-element `QgsFeatureRequest` (no layer mutation) | AuthManager + log sanitization | Borrow AuthManager pattern *if/when* you add DB connections. |
| **Window model** | standalone top-level window | modal dialog | ✅ Your non-modal window is the better UX. |
| **File hygiene** | many small modules (per your rules) | several 100 KB+ files | ✅ Keep your small-file discipline; don't copy their monoliths. |

### Recommended "imitate-and-improve" priorities
1. **Expand chart types via a declarative slot/registry** (their `SLOT_DEFINITIONS` idea, your clean small-file style). Highest value-per-effort.
2. **Add a pivot/matrix tile** using a **layer-backed, pandas-free** bucket engine (their `pivot_engine.py` shape, minus the pandas dependency) — keeps your zero-dependency edge while closing the biggest feature gap.
3. **Upgrade cross-filtering with directionality** and optional **on-canvas link wires** with orthogonal routing.
4. **Add multi-page dashboards** and **canvas zoom/pan + resize handles**.
5. **Add style presets + copy/paste tile styles** on top of your existing `Theme`.
6. **Offer exportable dashboards** (external JSON) in addition to `.qgz`-embedded layout, for sharing.
7. **Steal the safety patterns**: schema `version` + real migration on load, `MAX_RENDER_ITEMS` + signature-based repaint guards, and (when DB features land) AuthManager + log sanitization.
8. **Avoid their pitfalls**: don't add pandas, keep files small, move heavy aggregation to `QgsTask` from day one.

# Interactive HTML Export — Design Spec

**Date:** 2026-06-16
**Status:** Approved design, pending implementation plan
**Feature branch context:** `feature/multi-page-canvas-ux`

## 1. Goal

Let a user, after building a dashboard in the QGIS Dashboard plugin, **export it to a single
self-contained `index.html` file** they can double-click to open in any browser — fully offline,
with no framework, no Node, no server, and no internet connection.

The exported page reproduces the dashboard faithfully:

- multi-page layout with the tab bar, the grid placement of every tile, and the theme;
- **live cross-filtering** — clicking a chart bar / pie slice / pivot cell re-filters every connected
  tile on that page, exactly as in the plugin;
- charts, pivots, indicators, lists, text, and image tiles render with the dashboard's look.

The **map tile** is the one element that cannot be live in a double-click file (see §3.5): it is
exported as a **static PNG snapshot** of the QGIS map canvas.

## 2. Why a single vanilla-JS file (rejected alternatives)

The output is deliberately **one HTML file with everything inlined**, not a framework project.

- **Frameworks (Next.js/React) were considered and rejected for this goal.** A framework requires a
  build/serve step, which defeats the double-click-offline requirement. Vanilla JS provides all the
  interactivity needed (the cross-filter is plain data manipulation), so a framework adds cost without
  buying anything here.
- **The `file://` protocol constrains the design** and the single-file approach is what satisfies it:
  - `fetch()` of local files and ES-module `import`s are blocked under `file://` (CORS). → All CSS,
    JS, and data are **inlined** into `index.html`; scripts are classic `<script>`, not modules; data is
    embedded and read, never fetched.
  - Web Workers are blocked under `file://`. → No MapLibre / no library that needs a worker. The map is
    a static image (§3.5).

If the user opens the same file through any local static server, the limits vanish — but the artifact is
designed and tested for the literal double-click case.

## 3. Architecture

### 3.1 Output

A single file `<dashboard-name>.html` written to a user-chosen path. It contains, in order:

1. inlined `<style>` — structural CSS plus the theme expressed as CSS custom properties (`:root { --accent: …; }`);
2. the dashboard markup shell (tab bar + page containers);
3. `<script type="application/json" id="dashboard-data">` — the embedded export model (§3.3);
4. inlined `<script>` — the runtime (§3.4), read as classic script (no modules).

### 3.2 Faithful cross-filter (the core problem)

The browser must reproduce the plugin's `source → action → target` cross-filter **without** a
QgsExpression parser. The plugin's own semantics (read from `bus.py`, `chart.py`, `pivot_engine.py`)
are mirrored in JS:

- The **page-local connection graph** (`source_id → [target_ids]`) is exported verbatim from
  `DashboardBus.connections_to_dict(page_id)`.
- Source filters generated by the plugin are always **equality predicates** — `chart` pushes
  `"field" = value` (one `(field, value)` pair), `pivot` pushes a cell (two pairs) or a row/column
  header (one pair), `category_selector` pushes one pair. The runtime represents every active source
  selection as a list of `(field, value)` equality pairs and ANDs them — a direct port, not a guess.
- **`base_filter`** (an arbitrary, optional user QgsExpression) is **evaluated server-side at export
  time** through the existing `DashboardElement.iter_features()` path. Each tile stores the **indices of
  the feature rows that pass its `base_filter`**. The runtime never parses an expression — it intersects
  index sets. This keeps arbitrary expressions faithful with zero JS expression engine.

### 3.3 Embedded export model (`data.json`, inlined)

```jsonc
{
  "version": 1,                      // export-format version, independent of the v3 layout version
  "grid": { "cols": 12, "rows": 8 },
  "theme": { /* Theme.to_dict() */ },
  "active_page": "<id>",
  "layers": {
    "<layer_id>": {
      "fields": ["name", "pop", ...],          // all fields
      "features": [ { "name": "...", "pop": 12 }, ... ]   // all rows
    }
  },
  "pages": [
    {
      "id": "<id>", "title": "Page 1",
      "connections": { "<source_id>": ["<target_id>", ...] },
      "tiles": [
        {
          "id": "<id>", "type": "chart",
          "grid": { "x": 0, "y": 0, "w": 4, "h": 3 },
          "config": { /* whole element config incl. style override */ },
          "layer_id": "<layer_id>",            // null for text/image/map
          "base_pass": [0, 2, 5, ...],         // feature indices passing base_filter; null = all
          "map_image": "data:image/png;base64,..."  // map tiles only
        }
      ]
    }
  ]
}
```

Rows are stored **once per layer** and referenced by `layer_id`, not duplicated per tile. "All fields,
all rows" applies (per the chosen data scope), bounded by the size guard (§3.6).

### 3.4 Runtime (`assets/runtime.js`, inlined at build)

Vanilla JS, no dependencies. Responsibilities:

- **Filter engine** — mirror of `bus.combined_filter_for` + each source's click/toggle logic
  (`ChartElement._selected` semantics). Holds per-source selection state; on a source click, recomputes
  every connected target.
- **Aggregation** — port of `ChartElement._aggregate` (group-by + count/sum/mean, `fold_categories`
  "Other" folding from `chart_specs`) and `pivot_engine.compute_pivot` (row/col bucketing, totals,
  truncation).
- **Renderers per tile type:**
  - `chart` — inline **SVG** painters mirroring `elements/charts/painters.py`
    (bar / barh / line / area / pie / donut); clickable, drives the filter engine.
  - `pivot` — HTML table mirroring `pivot_engine`; clickable cells / row & column headers.
  - `indicator` — single aggregate value (+ optional reference/trend).
  - `list` — feature table; row selection highlights only (the map is static, so no live
    `featureAction` zoom).
  - `text` / `image` — presentational. The image tile's source file is **embedded as a base64 data
    URI** at export time (a referenced path would not travel with a double-click file); if the file is
    missing/unreadable, render the placeholder.
  - `map` — renders the embedded **static PNG snapshot**, scaled to the tile (`accepts_filter=False`,
    so it ignores filters, consistent with the plugin).

The runtime ships as a **real static file** in the plugin (`export/assets/runtime.js`), not a Python
string literal, and is inlined into `index.html` by the builder. Same for `export/assets/runtime.css`.

### 3.5 The map tile

A double-click `file://` page cannot run MapLibre (Web Worker blocked) and an online basemap needs
internet regardless. Therefore the map tile is exported as a **static image**: grab
`iface.mapCanvas()` to a `QImage`/`QPixmap`, encode PNG, base64-embed as the tile's `map_image`. This is
faithful to what the user currently sees (current extent, layers, basemap) and is fully offline.

### 3.6 Large-data guard

Because all rows are embedded, a large bound layer can bloat `index.html`. Before writing, estimate
output size (feature count × field count; rough byte estimate). If a layer crosses a threshold
(initial values: **> 100k features or > 50 MB estimated**), the export dialog shows a warning naming the
layer and offering **Proceed / Skip this layer / Cancel**, with wording that a large GPKG/Shapefile
embedded for browser viewing is not advisable. (Map snapshots are single bounded images and are exempt.)

### 3.7 Entry point

An **"Export to HTML…"** button in the existing `settings_dialog.py → SettingsDialog` hub (alongside
"Open Appearance…" and About). Per the codebase convention, infrequent actions live in the Settings hub,
not on the always-visible rail. It opens `export_dialog.py → ExportDialog` (output path picker + the
size warning), then calls the orchestrator.

## 4. Python module layout

New package `qgis_dashboard/export/`, small focused modules, most of them Qt-free and unit-testable
(following the `pivot_engine` precedent):

| Module | Purpose | QGIS/Qt? | Tested |
|---|---|---|---|
| `export/serialize.py` | Pure: window/page/tile/theme/graph → export-model dict | no | pytest (plain dicts) |
| `export/theme_css.py` | Pure: `Theme` → CSS custom-property block (web mirror of `window_qss`) | no | pytest |
| `export/data_collect.py` | Per-layer rows (all fields) + per-tile `base_filter` pass-index masks; size estimate | yes | thin; logic in pure helpers |
| `export/map_snapshot.py` | Grab `iface.mapCanvas()` → base64 PNG for map tiles | yes | manual |
| `export/html_builder.py` | Assemble final `index.html`: inline CSS + runtime JS + embedded JSON | no | pytest (string assembly) |
| `export/html_export.py` | Orchestrator: window state → written `index.html` | yes | manual |
| `export/assets/runtime.js` | Vanilla-JS runtime (filter engine, SVG painters, renderers) | n/a (web) | vitest (optional) + manual |
| `export/assets/runtime.css` | Structural CSS; theme injected as `:root` vars | n/a (web) | manual |
| `export_dialog.py` | Output path picker + large-data warning | yes | manual |

Wiring obligations (per project conventions):

- Register `export/*.py`, `export_dialog.py` in **both** `pb_tool.cfg` (`python_files`) and `Makefile`
  (`PY_FILES`).
- Register `export/assets/` as an `extra_dir` in **both** `pb_tool.cfg` (`extra_dirs`) and `Makefile`
  (`EXTRA_DIRS`), so the runtime files ship in the packaged zip.
- Relative imports within the package (`from .export.html_export import …`).
- Qt via `qgis.PyQt` only.

## 5. Data flow

```
DashboardWindow (current state: pages, tiles, bus, theme)
        │
        ▼
html_export.export(window, out_path)
        │  ├─ data_collect: per-layer rows (all fields) + per-tile base_filter pass-indices + size estimate
        │  ├─ map_snapshot: iface.mapCanvas() → base64 PNG (per map tile)
        │  ├─ serialize:    build export-model dict (layout + graph + layers + theme)
        │  ├─ theme_css:    Theme → :root CSS variables
        │  └─ html_builder: inline runtime.css + runtime.js + JSON(model) → index.html string
        ▼
write <dashboard-name>.html
```

Cross-filter at view time (in the browser, all client-side):

```
click on source tile (chart/pivot/selector)
  → set that source's selection (field,value pair[s])
  → for each target connected to active sources:
        rows = layer.features filtered to tile.base_pass
        rows = rows AND-filtered by every connected source's (field == value)
        re-aggregate + re-render the tile
```

## 6. Error handling & edge cases

- **No tiles / empty dashboard** → export a valid page with an empty-state message; never crash.
- **Tile bound to a missing/invalid layer** → emit the tile with empty data and a small "no data" note
  (mirrors plugin behavior where `layer()` is `None`).
- **`base_filter` that fails to evaluate server-side** → treat as "all rows pass" and log; do not abort
  the whole export.
- **Map tile with no map canvas / grab failure** → render a labeled placeholder instead of a broken image.
- **Output path not writable** → surface a clear dialog error; do not partially write.
- **Non-JSON-serializable config values** → coerce/validate in `serialize.py`; fail fast with a clear message.
- **Field name collisions / special characters** → values are embedded as JSON data (not built into
  expression strings), so quoting/escaping is handled by `json.dumps`.

## 7. Testing strategy

- **Pure Python** (`serialize`, `theme_css`, `html_builder`, and any extracted aggregation/predicate
  helpers): pytest on plain dicts, no QGIS runtime — the `pivot_engine` pattern.
- **Reference parity:** generate aggregation/pivot reference values from the Python tests and assert the
  JS `aggregate.js`/filter engine produces the same (vitest in the asset dir, optional but recommended).
- **QGIS-touching** (`data_collect`, `map_snapshot`, `html_export`, dialogs): manual verification inside
  QGIS — build a small multi-page dashboard with a chart→list connection and a map, export, double-click
  the file, confirm cross-filter works offline and the map snapshot shows.
- **Offline guarantee:** the acceptance test is literally double-clicking the file with networking off.

## 8. Out of scope (deliberate, may be follow-ons)

- Interactive (pan/zoom) web map — incompatible with the double-click/offline requirement; a separate
  "served/deployed" export mode could add MapLibre + a mirrored basemap later.
- Live `featureAction` map zoom from list selection (the map is a static image here).
- Re-querying the original data source after export (the export is a point-in-time snapshot of data).
- Any framework/build/deploy pipeline (Next.js, Vercel, etc.).

## 9. Acceptance criteria

1. "Export to HTML…" in Settings produces a single `.html` file at the chosen path.
2. Double-clicking that file (no server, networking disabled) opens the dashboard with correct theme,
   pages, tab bar, and tile placement.
3. Clicking a chart bar / pie slice / pivot cell live-filters every connected tile on the page, matching
   the plugin's behavior.
4. Charts are inline SVG and match the plugin's chart types and "Other"-folding.
5. The map tile shows a static snapshot of the QGIS canvas.
6. A layer exceeding the size threshold triggers the Proceed / Skip / Cancel warning.
7. New modules and `export/assets/` are registered in `pb_tool.cfg` and `Makefile`; `py_compile` of all
   modules passes.
